1 # SVEA-Arduino
2 The package contains the arduino code for the SVEA cars and various support functions.
3 The Arduino on the power board has three main tasks:
4 1. Recieve messages from the TX2 via a rosserial node and translate those into pwm-signals.
5 2. Read pwm-signals from the reciever and transfere the messages to the TX2.
6 3. Count the toicks from the wheel encoders and send the results to the TX2.
7 If the TX2 is not sending anything (or the remote is put into override mode by setting the switch into the forward position), the Arduino can also transmit the input from the remote directly to the pwm board without the TX2 having to be turned on.
8 
9 ## Quickstart
10 1. Turn on the remote.
11 2. Make sure that the LED on the reciever shines with a steady green light.
12 Run `roslaunch svea_arduino basic_remote_reader.launch`. If everything is turned on, installed and setup correctly, it should now be possible to control the car via the remote. The actuation values should also be published on the `/lli/ctrl_actuated` topic in ROS.

14 ## Messages
15 The package contains some custom messages.
16 ### lli_ctrl
17 These messages contains four fields. All of them are signed 8-bit integers (can contain values between 127 and -128).
18 The first two fields are `steering` and `velocity`. Steering goes in a positive clockwise direction, i.e. -127 is maximum left (about 45 degrees, pi/2 radians left) and 127 is maximum right (about 45 degrees, pi/2 radians right). Velocity also goes from -127, full brake/reverse to 127, full speed ahead. The reverse behavior depends on the setting of the ESC. By default a reverse signal will first make the car brake. If a neutral (zero) signal is then sent followed by another negative signal, the car will go into reverse.
19 Sending -128 to either steering or velocity will make the Arduino keep sending the last non -128 value to that channel.
20 
21 The `trans_diff` control the transmision and differentials on the car. The field is treated as a series of boolean (True/False) values, where every bit has a sepparate meaning according to:
22 
23 | Bit | Usage |
24 | --- | ----- |
25 | 0 | Transmision, 0 = low gear, 1 = high gear |
26 | 1 | Front differential, 0 = unlocked, 1 = locked |
27 | 2 | Rear differential, 0 = unlocked, 1 = locked |
28 | 3 | Change transmission according to bit 0 if this is set |
29 | 4 | Change front differential according to bit 1 if this is set |
30 | 5 | Change rear differential according to bit 2 if this is set |
31 
32 Note: The bits are zero indexed.
33 Bits 3, 4, and 5 are do not care bits and have the same role as -128 for steering and velocity. If these bits are set to 0, the Arduino will ignore the corresponding value.
34 For example: if a message is sent with bit 0 set to 1, requesting the high gear, but bit 3 is set to 0, the transmision will remain in watever gear it was in before the message was sent. However, if bit 1 is set to 1 and bit 4 is set to 1 in the same message, the front differential will be locked, but the transmision bit will still be ignored.
35 
36 Messages sent from the arduino to the TX2 will always have the do notcare bits set to 1.
37 
38 The `control` field is used to recieving (and in the future, sending) control flags from the Arduino. According to:
39 
40 | Bit | Meaning |
41 | --- | ------- |
42 | 0 | TX2 detected as idle (have not sent any messages for 5 seconds) |
43 | 1 | Remote disconnected (No pwm signal on channel 5 from the reciever) |
44 | 2 | Remote in override mode |
45 
46 The control field is currently only used for recieving messages and is ignored in messages sent to the Arduino.
47 
48 ### lli_encoder
49 Contains the number of ticks on the right wheel and left wheel during time delta. Note that the resolution of time delta is 10 micro seconds.
50 Time delta is currently the duration from the previous message being sent to the current message being sent. It will be changed to the duration
51 between the last tick counted in the previous message and the last tick counted in this message, and split ointo two fields, one for the left wheel and one for the right.

56 ### Arduino to PWM Board
57 Required for control of the car.
58 The pwm-board is the blue Adafruit board with a large capacitor and lots of pins for connecting servos.
59 
60 | Arduino pin | Connects to | Notes|
61 | --- | --- | --- |
62 | A4/SDA | SCA pin on pwm board | i2c buss data |
63 | A5/SCL | SCL pin on pwm board | i2c buss clock |
64 | 5V | VCC pin on pwm board | The VCC pin should be connected to 5V, it does not have to be connected to the 5V pin on the ardunio. |
65 | GND | GND pin on pwm board | The GND pin should be connected to ground, it does not have to be connected to the ground pin on the ardunio.|
66 | D3 | PWM pin 10 on the pwm board | Optional, used for adjusting the output frequency of the pwm board. |
67 
68 The internal clock on the pwm boards varries with up to 10% between boards. If the D3 pin is connnected the arduino will attempt to meassure the output frequency of the pwm board and adjust it to be close to 99 Hz. The servos work for higher frequencies, but the ESC will not recognize the signal if it is even slightly above 100 Hz.
69 The auto adjustment uses external interrupt 1 (INT1) on the arduino. The interrupt is disabled after the adjustment process is completed.

83 ### Reciever
84 Required connections for using the remote. If the remote is not going to be used pin D2 and D12 have to be connected to ground.
85 Only the pwm pin on the reciver should be connected unless otherwise noted.
86 
87 | Arduino pin | Connects to | Notes |
88 | --- | --- | --- |
89 | D2 | PWM output 1 (Steering) on reciever | Used for detecting the rising edge of the pwm signal |
90 | D8 | PWM output 1 (Steering) on reciever | There are 2 channel 1 pins on the receiver. Detects falling edge on channel 1. |
91 | D9 | PWM output 2 (ESC) on reciever | Detects falling edge on channel 2. |
92 | D10 | PWM output 3 (Gear) on reciever | Detects falling edge on channel 3. |
93 | D11 | PWM output 4 (Front ) on reciever | Detects falling edge on channel 4. |
94 | D12 | PWM output 5 (ESC) on reciever | Detects falling edge on channel 5. |
95 | 5V | Any power in pin on the reciever | The VCC pin should be connected to 5V, it does not have to be connected to the 5V pin on the ardunio. |
96 | Gnd | Any ground pin on the reciever | The GND pin should be connected to ground, it does not have to be connected to the ground pin on the ardunio.|
97 
98 The D2 pin uses external interrupt 0 (INT0) of the arduino. Pin D8 to D12 uses pin change interrupt 0 (PCINT0) and these pins are grouped on the GPIO adress PORTB.

100 ## Setup
101 Howto setup all the software inorder to make things work. This section does not include the standard ROS procedures for installing packages.
102 ### Changing the arduino device name
103 Required for running the default launch scripts.
104 In the default launch files the Arduino's device name is `/dev/arduinoPWM`. To create this link:
105 1. Make sure that the control Arduino is the only Arduino connected to the TX2 on the car
106 2. When in the package base folder, run `sudo python gen_arduino_rule.py`.
107 3. If no error messages appear the link should have been created. Check by typing `ls /dev/` and see if `/dev/arduinoPWM` is listed.
108 
109 ### Seting up arduino libraries
110 To compile and upload the firmware to the arduino some ROS libraries have to
111 be created by rosserial and put in the Arduino IDE library folder.
112 Custom messages requires this procedure everytime their definition is changed.
113 First run `catkin_make` and make sure that the messages are created correctly.
114 Then run `bash make_library.sh` from the `svea_arduino` directory.
115 (it will run `rm -r ~/Arduino/libraries/ros_lib/` followed by
116 `rosrun rosserial_arduino make_libraries.py /home/nvidia/Arduino/libraries/`)

118 ## Trouble shooting
119 
120 ### Rosserial reports that the Arduino is not found
121 1. Make sure that the Arduino is connected to the TX2 throught USB.
122 2. Type `ls /dev/` and check that `/dev/arduinoPWM` is listed. If not: check if `ttyACM0` is listed.
123 3. If `ttyACM0` is listed, then see `Changing the arduino device name` under `Setup`.'
124 
125 ### The LED on the ESC flashes with a green light and the car will not drive forward.
126 The blinking green light indicates that the ESC is not recieving a correct pwm signal.
127 1. Check that the ESC is connected to the pwm board via the servo cable. (Note that the power cable from the ESC should not be connected to the pwm board).
128 2. Check that pin 10 on the pwm board is connected to pin D3 on the Arduino. If the pin is disconnected: Connect it and restart the arduino.
129 3. If the problem persists: Recalibrate the ESC. See the TRX4 manual page 18 under "XL-5 HV Setup Programming" for how to do this.
130 
131 ### The car makes sudden sharp turns or accelerations
132 1. Make sure that the recievers is correctly connected to the Arduino
133 2. If no reciever is present, connect pin D2 and D12 to ground.
134 
135 ### Wheel encoders are registering different rates
136 1. Check that all cables from the sensors, to the sensor boards and
137 from the sensor boards to the wheel sensors are connected.
138 2. Check that the left LED och each sensor board shinse with a steady light.
139 If not: Check the connections to the power again.
140 3. Try turning the wheels by hand. The right LED on each sensor board should blink.
141  If not: Try adjusting the potentiometer on the sensor board.
142 4. If both right LEDs blink, run `roslaunch svea_arduino basic_remote_reader.launch`
143 and start `rosrun rqt_plot rqt_plot`. configure rqt_plot to show
144 `left_ticks` and `right_ticks` from the `/lli/encoder` topic. Try tweaking
145 the potentiometer on the sensorboard that appears to be giving eroneous readings
146 while using the remote to control the speed of the wheels.
